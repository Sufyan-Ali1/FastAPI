# Phase 1 Assignment 1B - Backend API Projects

> Goal: Master Phase 1 by building realistic backend APIs from a blank FastAPI project.
> Scope: Only use concepts from Phase 1 lessons 1-8.
> Standard: Portfolio-style backend practice, but implemented with beginner-safe Phase 1 tools.

---

## Allowed Phase 1 Concepts

Use only:

- `FastAPI()`
- `@app.get`, `@app.post`, `@app.put`, `@app.delete`
- Path parameters with type hints and `Path()`
- Query parameters with defaults, optional values, lists, and `Query()`
- Request bodies with Pydantic `BaseModel`
- Field validation with `Field()`
- Nested Pydantic models
- Combining path, query, and body parameters
- `Body()` and `embed=True` if needed
- Returning dictionaries, lists, and Pydantic models
- `status_code=...`
- `HTTPException`
- `JSONResponse` only when a normal `return dict` is not enough
- The injected `Response` object for dynamic status codes or simple headers
- JSON file storage using Python's standard `json` module
- Python dictionaries/lists while processing loaded JSON data
- Basic Python business logic

Do not use:

- Databases, SQLAlchemy, Alembic, SQLModel, MongoDB, Redis
- `APIRouter`
- Dependency injection with `Depends`
- Middleware
- Authentication, authorization, JWT, OAuth2, password hashing, sessions
- `response_model`
- Pydantic custom validators such as `@field_validator` or `@model_validator`
- Form data or file uploads
- Headers/cookies as request inputs
- Templates, static files, frontend pages
- Background tasks, WebSockets, caching, rate limiting
- Pytest or automated testing frameworks
- Docker, deployment, environment configuration

Important: Use local JSON files for storage. Data should remain available after the server restarts.

---

## General Rules For All Projects

- Build backend APIs only.
- Do not write a frontend.
- Do not copy solution code from the internet.
- Every project must run with `uvicorn main:app --reload`.
- Every resource should have a clear ID.
- Return proper HTTP status codes.
- Use `HTTPException` for errors such as not found, duplicates, invalid transitions, or business rule violations.
- Use Pydantic models for request bodies.
- Use `Field()`, `Path()`, and `Query()` validation wherever appropriate.
- Store and retrieve application data from JSON files.
- Load JSON data at the start of each request or through a small helper function.
- Save JSON data after every create, update, delete, or workflow action.
- Use simple `page` and `limit` query parameters only if you implement pagination. Do not use cursor pagination.
- Keep all endpoint implementations in `main.py`. Splitting into routers is a Phase 2 topic.

Recommended JSON storage rules:

- Put JSON files inside a `data/` folder.
- Keep each resource type in its own JSON file.
- Store records as a list of objects.
- Generate IDs by reading the existing records and using the next available integer.
- If a JSON file does not exist yet, create it with an empty list or seeded records.
- Do not use a database, ORM, or external storage service.
- Do not use concurrent file locking or advanced persistence patterns. This assignment is for local learning.
- Use helper functions such as `load_json(filename)` and `save_json(filename, data)` if you want to avoid repeated code.

Recommended pagination rules when used:

- `page`: default `1`, minimum `1`
- `limit`: default `10`, minimum `1`, maximum `100`
- Response should include `items`, `page`, `limit`, and `total`

# Project 1 - Neighborhood Services Directory API

## Difficulty

Medium

## Problem Statement

Build an API for a neighborhood services directory where users can discover local service providers such as plumbers, tutors, electricians, cleaners, mechanics, and designers.

The API should allow service providers to be created, searched, filtered, updated, reviewed, and removed. The goal is to practice resource design, path/query/body parameters, validation, and simple business rules.

## Functional Requirements

- Create service provider profiles.
- List providers with search, filtering, sorting, and pagination.
- Fetch one provider by ID.
- Fully update a provider profile.
- Delete a provider.
- Add reviews to providers.
- List reviews for one provider.
- Return provider summary information, including average rating and review count.

## API Requirements

- Store providers in `data/providers.json`.
- Store reviews in `data/reviews.json`.
- Load providers and reviews from JSON files when handling requests.
- Save providers and reviews back to JSON files after create, update, delete, and review actions.
- Provider IDs must be generated by the API.
- Review IDs must be generated by the API.
- Search should be case-insensitive.
- Filtering should happen through query parameters.
- Do not use authentication. Assume all requests are public.

## Required Endpoints

| Method | Path | Purpose |
|---|---|---|
| `POST` | `/providers` | Create a provider |
| `GET` | `/providers` | List/search/filter providers |
| `GET` | `/providers/{provider_id}` | Get one provider |
| `PUT` | `/providers/{provider_id}` | Replace provider details |
| `DELETE` | `/providers/{provider_id}` | Delete provider |
| `POST` | `/providers/{provider_id}/reviews` | Add a review |
| `GET` | `/providers/{provider_id}/reviews` | List reviews for a provider |
| `GET` | `/categories` | List available provider categories |

## Request And Response Expectations

`POST /providers` request should include:

- `name`
- `category`
- `city`
- `area`
- `phone`
- `hourly_rate`
- `years_experience`
- `available`
- `skills`

Successful `POST /providers` response:

- Status code `201`
- Created provider object
- Generated `id`
- `average_rating` initialized to `0`
- `review_count` initialized to `0`

`GET /providers` should support:

- `q`: optional search by provider name or skill
- `category`: optional category filter
- `city`: optional city filter
- `available`: optional boolean filter
- `min_rate`: optional minimum hourly rate
- `max_rate`: optional maximum hourly rate
- `min_experience`: optional minimum years of experience
- `page`: optional pagination page
- `limit`: optional pagination limit

Successful list response should include:

- `items`
- `page`
- `limit`
- `total`
- `filters_applied`

`POST /providers/{provider_id}/reviews` request should include:

- `reviewer_name`
- `rating`
- `comment`

Successful review response:

- Status code `201`
- Created review object
- Updated provider rating should be reflected when fetching the provider again

## Validation Requirements

- `provider_id`: integer, greater than or equal to `1`
- `name`: 2-80 characters
- `category`: one of a fixed set such as `plumber`, `electrician`, `cleaner`, `tutor`, `mechanic`, `designer`
- `city`: 2-60 characters
- `area`: 2-80 characters
- `phone`: must match a simple phone pattern, for example digits plus optional `+`, spaces, or dashes
- `hourly_rate`: greater than `0`
- `years_experience`: greater than or equal to `0`
- `skills`: list of strings, at least one skill required
- `reviewer_name`: 2-60 characters
- `rating`: integer from `1` to `5`
- `comment`: 5-500 characters
- `page`: greater than or equal to `1`
- `limit`: between `1` and `100`

## Business Rules

- Provider category must be from the allowed category list.
- Provider names do not need to be globally unique.
- A provider cannot be deleted if they have more than 10 reviews. Return a conflict error.
- Average rating must be recalculated after every new review.
- `min_rate` cannot be greater than `max_rate`.
- If no providers match a filter/search, return an empty `items` list, not an error.
- Deleting a provider should also remove or ignore its reviews in future review listing responses.

## Edge Cases

- Fetching a missing provider should return `404`.
- Updating a missing provider should return `404`.
- Adding a review to a missing provider should return `404`.
- Creating a provider with an invalid category should return validation or business-rule error.
- `GET /providers?available=maybe` should fail validation.
- `GET /providers?min_rate=100&max_rate=50` should return a clear error.
- Deleting the same provider twice should return `404` the second time.

## Expected Folder Structure

```text
neighborhood_services_api/
    main.py
    requirements.txt
    README.md
    data/
        providers.json
        reviews.json
```

## Deliverables

- A runnable FastAPI app.
- Swagger UI should show all endpoints clearly.
- A short README explaining how to run the app.
- At least 8 sample requests documented in the README.
- JSON seed data for at least 5 providers and 5 reviews.
- Data should still exist after stopping and restarting the server.

## Bonus Challenges

- Add `sort_by` query parameter with allowed values: `hourly_rate`, `years_experience`, `rating`.
- Add `order` query parameter with allowed values: `asc`, `desc`.
- Add a `GET /providers/top-rated` endpoint that returns providers with average rating greater than or equal to a query parameter.
- Add a `GET /providers/{provider_id}/summary` endpoint that returns only summary fields.

# Project 2 - Inventory And Purchase Order API

## Difficulty

Hard

## Problem Statement

Build an inventory API for a small warehouse that manages products, suppliers, and purchase orders.

This project simulates common backend interview problems: unique fields, stock calculations, nested request bodies, filtering, order state changes, and conflict handling.

## Functional Requirements

- Create and manage products.
- Create and manage suppliers.
- List products with filtering/searching/pagination.
- Create purchase orders containing multiple line items.
- Receive purchase orders and update stock.
- Cancel purchase orders.
- Track low-stock products.
- Calculate order totals from product prices and quantities.

## API Requirements

- Store products in `data/products.json`.
- Store suppliers in `data/suppliers.json`.
- Store purchase orders in `data/purchase_orders.json`.
- Load data from JSON files when handling requests.
- Save data back to JSON files after create, update, delete, receive, and cancel actions.
- Do not accept client-provided IDs.
- `sku` must be unique across products.
- Purchase orders must contain at least one item.
- Order item product IDs must reference existing products.
- Supplier IDs must reference existing suppliers.

## Required Endpoints

| Method | Path | Purpose |
|---|---|---|
| `POST` | `/products` | Create a product |
| `GET` | `/products` | List/search/filter products |
| `GET` | `/products/{product_id}` | Get one product |
| `PUT` | `/products/{product_id}` | Replace a product |
| `DELETE` | `/products/{product_id}` | Delete a product |
| `POST` | `/suppliers` | Create a supplier |
| `GET` | `/suppliers` | List suppliers |
| `GET` | `/suppliers/{supplier_id}` | Get one supplier |
| `POST` | `/purchase-orders` | Create a purchase order |
| `GET` | `/purchase-orders` | List purchase orders |
| `GET` | `/purchase-orders/{order_id}` | Get one purchase order |
| `POST` | `/purchase-orders/{order_id}/receive` | Receive an order and increase stock |
| `POST` | `/purchase-orders/{order_id}/cancel` | Cancel an order |
| `GET` | `/inventory/low-stock` | List low-stock products |

## Request And Response Expectations

`POST /products` request should include:

- `name`
- `sku`
- `category`
- `unit_price`
- `stock_quantity`
- `reorder_level`
- `active`

Successful product response:

- Status code `201`
- Generated `id`
- All submitted product fields

`GET /products` should support:

- `q`: optional search by product name or SKU
- `category`: optional filter
- `active`: optional boolean filter
- `low_stock`: optional boolean filter
- `min_price`: optional minimum price
- `max_price`: optional maximum price
- `page`
- `limit`

`POST /suppliers` request should include:

- `name`
- `contact_email`
- `phone`
- `city`
- `active`

`POST /purchase-orders` request should include:

- `supplier_id`
- `items`: list of order items
- `notes`: optional text

Each order item should include:

- `product_id`
- `quantity`

Successful purchase order response:

- Status code `201`
- Generated `id`
- `status` set to `draft`
- `items` with product IDs, quantities, and calculated line totals
- `total_amount`

`POST /purchase-orders/{order_id}/receive` response:

- Status code `200`
- Updated order with `status` set to `received`
- Product stock quantities increased by order quantities

`POST /purchase-orders/{order_id}/cancel` response:

- Status code `200`
- Updated order with `status` set to `cancelled`

## Validation Requirements

- `product_id`, `supplier_id`, `order_id`: integer, greater than or equal to `1`
- `name`: 2-100 characters
- `sku`: 3-30 characters, uppercase letters, numbers, and dashes only
- `category`: 2-50 characters
- `unit_price`: greater than `0`
- `stock_quantity`: greater than or equal to `0`
- `reorder_level`: greater than or equal to `0`
- `contact_email`: simple email pattern
- `phone`: simple phone pattern
- `city`: 2-60 characters
- `quantity`: integer, greater than `0`
- `notes`: optional, maximum 500 characters
- `page`: greater than or equal to `1`
- `limit`: between `1` and `100`

## Business Rules

- Duplicate SKU should return `409`.
- Product cannot be deleted if it appears in an existing purchase order.
- Inactive suppliers cannot be used for new purchase orders.
- Inactive products cannot be added to purchase orders.
- Purchase order total must be calculated by the server, not trusted from the client.
- A draft order can be received.
- A draft order can be cancelled.
- A received order cannot be cancelled.
- A cancelled order cannot be received.
- Receiving an order should update stock only once.
- `low_stock=true` should return products where `stock_quantity <= reorder_level`.
- `min_price` cannot be greater than `max_price`.

## Edge Cases

- Creating a product with an existing SKU should return `409`.
- Creating an order with missing product IDs should return `404`.
- Creating an order with a missing supplier ID should return `404`.
- Receiving a missing order should return `404`.
- Receiving an already received order should return `409`.
- Cancelling an already received order should return `409`.
- Listing products when none exist should return an empty list.
- Deleting a missing product should return `404`.
- Invalid query values should produce FastAPI validation errors.

## Expected Folder Structure

```text
inventory_purchase_order_api/
    main.py
    requirements.txt
    README.md
    data/
        products.json
        suppliers.json
        purchase_orders.json
```

## Deliverables

- A runnable FastAPI app.
- At least 6 seeded products.
- At least 3 seeded suppliers.
- At least 2 seeded purchase orders.
- Seed data should be stored in JSON files.
- Data should still exist after stopping and restarting the server.
- README with request examples for product creation, filtering, order creation, receiving, and cancelling.
- Clear error responses for missing resources and invalid state transitions.

## Bonus Challenges

- Add `GET /inventory/valuation` that returns total inventory value.
- Add `GET /products/{product_id}/movement-summary` that shows starting stock, ordered quantity, received quantity, and current stock using only JSON-stored data.
- Add `GET /purchase-orders?status=draft` filtering.
- Add query sorting for products by `name`, `unit_price`, or `stock_quantity`.

Advanced

Build a backend API for a helpdesk system where customers submit support tickets and support agents manage ticket status, priority, and comments.

This project focuses on real-world workflows, nested data, status transitions, filtering, searching, and strict validation without using authentication or a database.

## Functional Requirements

- Create customer records.
- Create support tickets for customers.
- List tickets with filters, search, and pagination.
- Fetch ticket details.
- Replace ticket details.
- Add internal or public comments to tickets.
- Change ticket status through controlled workflow endpoints.
- Assign tickets to support agents by name.

- Store tickets in `data/tickets.json`.
- Load customers and tickets from JSON files when handling requests.
- Save customers and tickets back to JSON files after create, update, delete, comment, assignment, and workflow actions.
- Use nested Pydantic models for customer contact details and ticket comments.
- No authentication. Treat `agent_name` as a normal request field where needed.
- Ticket IDs and customer IDs must be generated by the API.
- Ticket status must be controlled by business rules.

## Required Endpoints

| Method | Path | Purpose |

| `POST` | `/customers` | Create a customer |
| `GET` | `/customers` | List/search customers |

- `email`
- `phone`
- `company`
- `contact_address` as a nested object

- `city`
- `country`
- `postal_code`

- Customer details

`POST /tickets` request should include:

- `customer_id`
- `subject`
- `description`
- `category`
- `priority`

Successful ticket response:

- Status code `201`
- Generated `id`
- `status` set to `open`
- `comments` initialized as empty list
- `assigned_agent` initialized as `null` or equivalent empty value

`GET /tickets` should support:

- `q`: optional search by subject or description

- `customer_id`, `ticket_id`: integer, greater than or equal to `1`
- Customer `name`: 2-100 characters

- Customer `phone`: simple phone pattern

- `city`: 2-80 characters
- `country`: 2-80 characters

- Resolving an already resolved ticket should return `409`.
- Reopening an open ticket should return `409`.
- Assigning a resolved ticket should return `409`.
- Deleting a ticket with comments should return `409`.
- Listing tickets with filters that match nothing should return an empty list.
- Invalid status or priority query values should fail validation.

# Project 4 - Restaurant Ordering And Kitchen Workflow API

## Difficulty

Very Advanced

## Problem Statement

Build a backend API for a restaurant that manages menu items, dining tables, customer orders, kitchen workflow, and receipts.

This is the most complex Phase 1 project. It requires careful REST design, nested request bodies, query filters, calculated fields, state transitions, and many business rules while still using only Phase 1 concepts.

## Functional Requirements

- Create and manage menu items.
- List menu items with filters/search/pagination.
- Create dining tables.
- Open customer orders for tables.
- Add items to an order.
- Replace an order before it is submitted.
- Submit an order to the kitchen.
- Move order through kitchen statuses.
- Cancel orders when allowed.
- Generate an order receipt.
- List orders by table, status, or total range.

## API Requirements

- Store menu items in `data/menu_items.json`.
- Store tables in `data/tables.json`.
- Store orders in `data/orders.json`.
- Load menu items, tables, and orders from JSON files when handling requests.
- Save data back to JSON files after create, update, delete, submit, prepare, serve, and cancel actions.
- Menu item IDs, table IDs, and order IDs must be generated by the API.
- Prices and totals must be calculated by the server.
- Order item request bodies should use nested Pydantic models.
- No authentication, no payments integration, no database.
- No partial update endpoints. Use `PUT` for full replacement where required.

## Required Endpoints

| Method | Path | Purpose |
|---|---|---|
| `POST` | `/menu-items` | Create a menu item |
| `GET` | `/menu-items` | List/search/filter menu items |
| `GET` | `/menu-items/{item_id}` | Get one menu item |
| `PUT` | `/menu-items/{item_id}` | Replace a menu item |
| `DELETE` | `/menu-items/{item_id}` | Delete a menu item |
| `POST` | `/tables` | Create a dining table |
| `GET` | `/tables` | List tables |
| `GET` | `/tables/{table_id}` | Get one table |
| `POST` | `/orders` | Open a new order |
| `GET` | `/orders` | List/search/filter orders |
| `GET` | `/orders/{order_id}` | Get one order |
| `PUT` | `/orders/{order_id}` | Replace draft order items |
| `POST` | `/orders/{order_id}/submit` | Submit draft order to kitchen |
| `POST` | `/orders/{order_id}/prepare` | Mark submitted order as preparing |
| `POST` | `/orders/{order_id}/serve` | Mark prepared order as served |
| `POST` | `/orders/{order_id}/cancel` | Cancel order when allowed |
| `GET` | `/orders/{order_id}/receipt` | Get calculated receipt |

## Request And Response Expectations

`POST /menu-items` request should include:

- `name`
- `category`
- `description`
- `price`
- `available`
- `tags`

Successful menu item response:

- `status` set to `draft`

## Validation Requirements

- `item_id`, `table_id`, `order_id`: integer, greater than or equal to `1`
- Menu item `name`: 2-100 characters
- `category`: one of `starter`, `main`, `dessert`, `drink`, `side`
- `description`: 5-500 characters
- `price`: greater than `0`
- `tags`: list of strings
- `table_number`: integer, greater than or equal to `1`
- `capacity`: integer from `1` to `20`
- `section`: one of `indoor`, `outdoor`, `vip`, `bar`
- `customer_name`: optional, 2-80 characters when provided
- `menu_item_id`: integer, greater than or equal to `1`
- `quantity`: integer from `1` to `20`
- `special_instructions`: optional, maximum 200 characters
- `status`: one of `draft`, `submitted`, `preparing`, `served`, `cancelled`
- `page`: greater than or equal to `1`
- `limit`: between `1` and `100`
- `min_price` cannot be greater than `max_price`
- `min_total` cannot be greater than `max_total`

## Business Rules

- Orders must reference an existing table.
- Order items must reference existing menu items.
- Unavailable menu items cannot be added to new orders.
- Orders must contain at least one item.
- Draft orders can be replaced with `PUT`.
- Submitted, preparing, served, and cancelled orders cannot be replaced.
- Draft orders can be submitted.
- Submitted orders can move to preparing.
- Preparing orders can move to served.
- Draft and submitted orders can be cancelled.
- Preparing or served orders cannot be cancelled.
- A table becomes occupied when an order is submitted.
- A table becomes available again when an order is served or cancelled.
- A table cannot have two active submitted/preparing orders at the same time.
- Menu items used in non-cancelled orders cannot be deleted.
- Subtotal is the sum of item quantity multiplied by menu item price.
- Service charge is 10 percent of subtotal.
- Total is subtotal plus service charge.
- Totals must be calculated by the API, not accepted from the client.

## Edge Cases

- Creating an order for a missing table should return `404`.
- Creating an order with a missing menu item should return `404`.
- Creating an order with an unavailable menu item should return `409`.
- Submitting a missing order should return `404`.
- Submitting an already submitted order should return `409`.
- Preparing a draft order should return `409`.
- Serving a submitted order before preparing should return `409`.
- Cancelling a preparing or served order should return `409`.
- Replacing a non-draft order should return `409`.
- Deleting a menu item used by an active order should return `409`.
- Listing orders with filters that match nothing should return an empty list.
- Invalid boolean, integer, or enum-like query values should fail validation.
